# LLM coordinator trials (OpenAI)

This document describes the artifact produced when running the full LLM coordinator trials script with real OpenAI API for the hospital-lab pipeline.

## What is run

The script [scripts/run_llm_coord_trials_openai.py](../../scripts/run_llm_coord_trials_openai.py) runs **coord_scale** (default) or **coord_risk** (with `--task coord_risk --injection <id>`) with each of the four LLM coord methods:

- llm_central_planner
- llm_auction_bidder
- llm_central_planner_debate
- llm_central_planner_agentic

Each run uses **pipeline_mode=llm_live**, **llm_backend=openai_live**, **allow_network=True**, and a shared scale config (default: small_smoke; auction_bidder gets round_robin via that scale). The same lab policies (RBAC, shield, invariants) apply as in the deterministic pipeline.

## Requirements

- **OPENAI_API_KEY** must be set; the script exits with a clear error if not.
- Set **LABTRUST_LLM_TRACE=1** (or use `--trace`) to include attribution (by_backend call_count, latency_ms_sum, cost_usd_sum) in the report. Attribution (by_backend, cost, latency) is populated only when LABTRUST_LLM_TRACE=1; see [Observability](observability.md).

## How to reproduce

From the repo root:

```bash
LABTRUST_LLM_TRACE=1 python scripts/run_llm_coord_trials_openai.py --out-dir labtrust_runs/llm_coord_trials_openai
```

Optional: `--episodes 2`, `--seed 100`, `--methods "llm_central_planner,llm_auction_bidder"` to restrict methods; `--task coord_risk --injection INJ-BID-SPOOF-001` for risk trials.

## Output layout

| Path | Description |
|------|-------------|
| `<out_dir>/<method_id>_results.json` | Full benchmark results (results.v0.2) for that method. |
| `<out_dir>/llm_coord_trials_report.json` | Aggregated summary (see schema below). |
| `<out_dir>/llm_coord_trials_report.md` | Human-readable summary and reproduce command. |
| `<out_dir>/llm_coord_trials_interpretation.md` | Interpretation template (metrics + placeholder Conclusions) for publishing a run. |

## Report schema (llm_coord_trials_report.json)

- **scenario:** string (e.g. "coord_scale LLM coordinator trials (openai_live)" or "coord_risk ... injection=INJ-BID-SPOOF-001").
- **task:** string; "coord_scale" or "coord_risk".
- **scale_id:** string (e.g. "small_smoke").
- **base_seed:** int.
- **attribution_present:** bool; true when LABTRUST_LLM_TRACE was set.
- **injection_id:** string (optional); present when task is coord_risk.
- **methods:** array of objects, one per method:
  - **method_id:** string.
  - **task:** string (e.g. "coord_scale" or "coord_risk").
  - **num_episodes:** int.
  - **pipeline_mode:** string (e.g. "llm_live").
  - **llm_backend_id:** string or null.
  - **run_duration_wall_s:** float or null (from metadata).
  - **by_backend:** object or null; when attribution_present, keys are backend_id, values are { call_count, latency_ms_sum, cost_usd_sum }.
  - **injection_id:** string (optional); present when task is coord_risk.

## Interpreting results

The script and CI produce raw artifacts (`llm_coord_trials_report.json`, `llm_coord_trials_report.md`). To publish a summarized interpretation of a run (e.g. for a release or study), use the following structure.

**Scenario:** State the run context: coord_scale, pipeline_mode=llm_live, scale id, episodes per method, base seed, and whether attribution was collected (LABTRUST_LLM_TRACE=1).

**Metrics summary:** From `llm_coord_trials_report.json`, summarize per method: `run_duration_wall_s`, `num_episodes`, and when attribution is present: `by_backend` (call_count, latency_ms_sum, cost_usd_sum). Optionally include success/completion from per-method `*_results.json` (e.g. no error field, num_episodes completed).

**Conclusions:** Short narrative: relative cost and latency across methods, any caveats (scale, model, seed), and how to reproduce. This section is for human editing when publishing a specific run.

**Artifacts:** Reference `llm_coord_trials_report.json`, `llm_coord_trials_report.md`, and optionally `llm_coord_trials_interpretation.md` (if generated by the script). Reproduce command: same as in [How to reproduce](#how-to-reproduce) with the actual `--out-dir` used.

The script writes `llm_coord_trials_interpretation.md` by default with the above template filled from the report and a placeholder "Conclusions" section for you to complete when publishing a run. By default the script and CI produce raw artifacts (JSON + MD); the interpretation document addresses the need for a written summary of metrics and conclusions.

## coord_risk and trials

The trials script supports both **coord_scale** (default) and **coord_risk**. Use `--task coord_risk` together with `--injection <injection_id>` (e.g. `INJ-BID-SPOOF-001`, `INJ-COMMS-DELAY-001`) to run the same LLM coord methods under a risk injection. The report schema includes `task` and per-method `injection_id` when applicable; results conform to results.v0.2 and can include coordination and security metrics. This covers the full hospital-lab pipeline (scale + risk) with real API. Example:

```bash
LABTRUST_LLM_TRACE=1 python scripts/run_llm_coord_trials_openai.py --task coord_risk --injection INJ-BID-SPOOF-001 --methods llm_central_planner --episodes 1
```

Optional follow-up: run a full matrix (multiple injections, multiple methods) or integrate coord_risk into CI; the script and report already support single-task, single-injection runs.

## State of the art and limitations

**Current position:** The pipeline provides production-grade single-model and multi-role coordination (per-role backends, guardrails, attribution), coord_scale and coord_risk trials with real API, and a written interpretation template for publishing metrics. The same lab policies (RBAC, shield, invariants) apply to deterministic and LLM coordination.

**Limitations (research SOTA):** Debate aggregation is majority by default; set `coord_debate_aggregator: llm` and provide an aggregator backend (e.g. same as proposer with a generate method, or params.aggregator_backend) to use an LLM to merge proposals (falls back to majority on failure). Round_robin is N bidder calls per step then merge, not multi-round negotiation or message-passing between separate agent LLMs. The interpretation document (llm_coord_trials_interpretation.md) addresses the need for a written summary of cost, latency, and conclusions; complete it when publishing a run. See [Multi-LLM protocols](../coordination/multi_llm_protocols.md) for protocol-level limits.
